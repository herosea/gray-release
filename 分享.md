## 引言

欢迎大家今天来听我的分享，我今天给大家分享下codex的使用，以及Agentic Engineering。Agentic Engineering是2026年2月5日[Karpathy在vibe coding 一周年之际提出的概念](https://x.com/karpathy/status/2019137879310836075)，是Vibe Coding的进化概念。主要是去年AI编码能力的极大进步，生成的代码的质量足以大规模达到生产环境标准。
如果说Vibe Coding 是“全靠直觉和运气在调戏AI”，那么Agentic Engineering 就是“有组织、有纪律的特种兵作战”。
Agentic 有两个词组成
“Agentic”指有agent 或多agent系统来写代码，人类开发者负责监督和验证输出，保持human-in-the-loop；重点在人的职责变化。
“Engineering” 指使用agentic工作流进行有意义的代码生产需要一定的专业能力，而这种能力是可以培养和提升的。重点在工程的科学性和艺术性。

# 验证 npm 版本：
npm -v # Should print "11.9.0".

# 安装code

```

## Agentic Engineering

karpathy举了个例子
```
举个例子，上周末我在为家里的摄像头搭建本地视频分析仪表盘，于是写道：“这是我的 DGX Spark 的本地 IP 和用户名/密码。登录，设置 ssh 密钥，配置 vLLM，下载并测试 Qwen3-VL，搭建用于视频推理的服务器端点，做一个基础的网页 UI 仪表盘，测试所有功能，用 systemd 配置服务，为你自己记录内存使用笔记，并给我写一份 markdown 报告。”这个智能代理运行了大约 30 分钟，遇到了多个问题，在线研究解决方案，逐一解决，编写代码，进行测试，调试，设置服务，最后带着报告返回——一切就这么完成了。我全程没有碰任何东西。所有这些在仅仅三个月前可能还是一个轻松的周末项目，但今天你只需启动它，然后忘掉 30 分钟就好。
```
这个例子的重点在于写代码、调试、测试、设置服务、生成报告都是都是的Agent完成的，人只启动了他，给了必要的信息，然后去干其他事情。
[Karpathy感慨“品味”和“判断”是最后的瓶颈点](https://x.com/karpathy/status/2026731645169185220)。AI可以吐出10中实现方案，但哪一种是优雅的、可扩展的、符合用户直觉的？这取决于你的品味。当“写”代码不再难，“选”什么代码就成了核心竞争力。

### 环境反馈闭环

构建反馈闭环能让Agent自动工作。例如写测试用例并运行测试，Agent会自动修复通不过的测试用例。让codex编译和运行代码也能处理编译和启动的错误。
目标清晰，能收集反馈进行验证的任务codex干的比较好。
给codex足够的权限，给codex配置更多的传感器（例如安装必要的命令行命令），安装可以操作浏览器等的skill，codex会更聪明和能力更强
### 架构化上下文
1. 建立“Plan.md”驱动的执行流
	- 例如生成技术设计文档
	- 任务计划，实现的步骤，第一步实现啥，第二步，第三步
2. “Token层次结构“管理
	- Agent.md 项目长期记忆，重复出现的问题加进去
	- 上下文压缩策略，这个主要Agent来完成，我们能用的命令/compact /new 
3. 部署"自愈型"脚本
	* 给脚本加上纠错逻辑，出错了，唤醒Agent 去查日志等纠错

### HBS（高度定制化软件）
[Karpathy对于HBS的方向性洞察](https://x.com/karpathy/status/2024583544157458452)，内容中例子如下

以今早为例——最近我对有氧运动有些松懈，所以决定进行一个更严肃、有计划的实验，试图在 8 周内将静息心率从 50 降至 45。主要方法是通过设定每周 Zone 2 有氧运动和 1 次 HIIT 的总分钟数目标来实现。  

一小时后，我即兴编写了这个超级定制的仪表盘，专门用于这个特定实验，以跟踪我的进展。Claude 不得不逆向工程 Woodway 跑步机的云端 API 来提取原始数据，然后进行处理、过滤、调试，并创建了一个 Web 前端界面来跟踪实验。整个过程并非一帆风顺，我不得不发现并请求修复一些 bug，比如它搞混了公制和英制单位，以及在日历中将日期与天数匹配时出错等。

核心观点
1. **样本量 N=1**：HBS 是为了解决你个人的**特定、临时、极其琐碎**的需求而生的
2. **LLM 胶水 (LLM Glue)**：软件不再是下载一个 .exe 或 .apk，而是由 AI 充当“胶水”，把各种 **传感器（Sensors）** 和 **执行器（Actuators）** 粘合在一起，形成一个临时的、功能极度聚焦的 App。
3. **App Store 的终结**：他直言不讳地指出，传统的 **App Store** 模式（一堆独立的 App 供你选择）正在变得过时。未来的软件是**按需生成（Ephemeral Apps）**的。
